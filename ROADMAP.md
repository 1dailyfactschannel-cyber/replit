# Roadmap: Рекомендуемые улучшения TeamSync

## Дата анализа: 2026-04-21

---

## Часть 1. Высокий приоритет

### 1.1. Глобальный поиск
**Статус:** Не реализован  
**Описание:** Сейчас поиск только по сотрудникам (`/api/users/search`). Необходим единый поиск по задачам, проектам, чатам, комментариям, файлам.  
**ROI:** Максимальный — пользователи ежедневно теряют время на навигацию.

### 1.2. Зависимости задач
**Статус:** Не реализован  
**Описание:** Нет связей между задачами: «блокирует», «заблокирована», «родительская», «дочерняя» (кроме subtasks). Нужна таблица `task_dependencies`.  
**ROI:** Высокий — критично для планирования и отслеживания критического пути.

### 1.3. Вехи (Milestones / Releases)
**Статус:** Не реализован  
**Описание:** У проектов нет контрольных точек. Нельзя сгруппировать задачи под релиз, спринт или дедлайн.  
**ROI:** Высокий — необходимо для roadmap и планирования.

### 1.4. Wiki / База знаний
**Статус:** Не реализован  
**Описание:** Нет централизованного хранения документации команды. Вся информация размазана по чатам и комментариям.  
**ROI:** Высокий — снижает потерю знаний при уходе сотрудников.

### 1.5. Экспорт данных
**Статус:** Не реализован  
**Описание:** Нет выгрузки задач, отчётов, списка сотрудников в CSV / PDF. Блокирует интеграции и бэкапы.  
**ROI:** Средний — важно для enterprise-клиентов.

### 1.6. Автоматизация (Automation Rules)
**Статус:** Не реализован  
**Описание:** Нет триггеров: «когда задача перешла в Done → уведомить ответственного», «просрочена → повысить приоритет», «создана → назначить исполнителя по умолчанию».  
**ROI:** Высокий — масштабирует администрирование без роста нагрузки на админов.

## Часть 2. Средний приоритет

### 2.1. Gantt / Timeline
**Статус:** Не реализован  
**Описание:** Только Kanban-вью. Нет временной шкалы проекта с датами начала/окончания задач.  
**Аудитория:** Менеджеры проектов, руководители.

### 2.2. Спринты и бэклог (Agile)
**Статус:** Не реализован  
**Описание:** Для Agile-команд нужно: планирование спринтов, backlog, velocity, burndown-chart.  
**Аудитория:** Scrum-команды.

### 2.3. Нагрузка команды (Workload)
**Статус:** Не реализован  
**Описание:** Нет визуализации загруженности исполнителей. Сложно понять, кто перегружен, а кто свободен.  
**Аудитория:** Team Lead'ы, PM.

### 2.4. Повторяющиеся задачи
**Статус:** Не реализован  
**Описание:** Ежедневные/еженедельные задачи приходится создавать вручную. Нужен механизм `recurring_tasks`.  
**Аудитория:** Все пользователи.

### 2.5. Публичные ссылки
**Статус:** Не реализован  
**Описание:** Нельзя поделиться проектом/доской/задачей с внешними людьми без регистрации.  
**Аудитория:** Клиенты, стейкхолдеры.

### 2.6. Двухфакторная аутентификация (2FA / TOTP)
**Статус:** Не реализован  
**Описание:** У владельца и администраторов нет 2FA. Критично для безопасности.  
**Аудитория:** Владелец, администраторы.

### 2.7. API + Webhooks
**Статус:** Не реализован  
**Описание:** Нет внешнего API для интеграций. Нет webhook'ов для событий (новая задача, изменение статуса).  
**Аудитория:** Интеграторы, enterprise.

## Часть 4. Технические риски (исправляются)

### 4.1. Смешанное хеширование паролей [ИСПРАВЛЕНО]
- **Проблема:** `auth.ts` использует `scrypt`, `routes.ts` использует `bcrypt`
- **Исправление:** `hashPassword()` в `auth.ts` теперь использует `bcrypt`. `comparePasswords()` поддерживает оба формата для обратной совместимости.
- **Файлы:** `server/auth.ts`

### 4.2. Отсутствие валидации входных данных [ЧАСТИЧНО]
- **Проблема:** Большинство endpoint'ов не валидируют тело запроса
- **Исправление:** Добавлены базовые проверки на критичных endpoint'ах. Полная Zod-валидация рекомендуется в рамках отдельного рефакторинга.

### 4.3. Отсутствие rate limiting [ИСПРАВЛЕНО]
- **Проблема:** Только `/api/login` защищён от перебора
- **Исправление:** Добавлены `mutationLimiter` (20 мутаций/мин) и `registerLimiter` (3 регистрации/час). Применены к созданию пользователей, смене ролей, сбросу пароля, смене владельца, удалению ролей.
- **Файлы:** `server/routes.ts`, `server/auth.ts`

### 4.4. Нет audit log [ИСПРАВЛЕНО]
- **Проблема:** Нет централизованного лога действий администраторов
- **Исправление:** Создана таблица `audit_logs` с индексами. Добавлен `logAudit()` helper. Логируются: создание/удаление пользователей, смена ролей, удаление ролей, сброс/смена пароля, передача прав владельца.
- **Файлы:** `shared/schema.ts`, `migrations/0019_add_audit_logs.sql`, `server/routes.ts`

### 4.5. Graceful degradation Redis [НАБЛЮДЕНИЕ]
- **Проблема:** При падении Redis включается in-memory cache, но без механизма восстановления
- **Решение:** Добавить health-check и автопереподключение с exponential backoff

## Рекомендуемый порядок внедрения

1. **Глобальный поиск** — максимальный ROI, пользователи сразу почувствуют
2. **Зависимости задач + Вехи** — закрывает основной функциональный пробел
3. **Wiki** — снижает нагрузку на чаты
4. **Автоматизация** — масштабирует администрирование
5. **Gantt** — востребовано руководством
